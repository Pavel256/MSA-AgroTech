# ADR-002: Спецификация ограниченных контекстов, интеграций и стек технологий для MVP Животноводства X

**Статус:** ⚠️ На рассмотрении  
**Участники:** Архитектор платформы Животноводства X, DevOps-инженер, Руководитель ИИ-разработки  
**Дата:** 2026-06-05  

---

## 1. Context (Контекст)
При проектировании контейнерного уровня (C2) платформы «Животноводство X» возникла необходимость декомпозировать доменную бизнес-логику системы на изолированные **ограниченные контексты (Bounded Contexts)**. Это позволит спроектировать конкретные микросервисы для их последующей независимой разработки разными командами. 

Также требуется определить и обосновать технологический стек для локального транспорта сообщений (брокер очередей), локальной СУБД, среды контейнеризации и оркестрации на ферме, зафиксировать возникшие архитектурные компромиссы и риски интеграционных подходов для двух топологических решений (Heavy Edge и Thin Edge).

---

## 2. Требования

### Функциональные
В соответствии с методологией **FURPS+**:


| Категория | Требование |
| :--- | :--- |
| **F**unctional | **Изоляция доменов:** Разделение системы на независимые программные контейнеры (видеоаналитика, сбор датчиков, обработка правил, синхронизация данных).<br>**Асинхронный транспорт:** Обеспечение отказоустойчивого обмена сообщениями между локальными компонентами внутри фермы.<br>**API-Интеграция:** Предоставление документированных REST API / gRPC контрактов для интеграции с существующими приложениями холдинга (React-портал, Flutter-приложение, Power BI). |
| **U**sability | **Единое окно управления:** Новые сервисы животноводства должны отдавать данные в существующие интерфейсы холдинга без необходимости переключения пользователя между разными приложениями. |

### Нефункциональные


| Категория | Требование | Критичность |
| :--- | :--- | :--- |
| **Автономность шины** | Локальная очередь сообщений на ферме должна полноценно функционировать и доставлять алерты исполнителям при полном отключении внешнего интернета. | **Критическая** |
| **Низкое потребление ресурсов** | Локальный программный стек на ферме (кроме GPU под видеоаналитику) должен быть не требователен к ресурсам сервера. | **Высокая** |
| **Сквозная задержка (SLA)** | Внутри локального сервера время от фиксации инцидента сервисом ИИ до вычитки движком правил и отправки команды на реле не должно превышать 1 секунды. | **Критическая** |
| **Оркестрация и развертывание** | Технологии локального сервера должны поддаваться простой автоматической кластеризации и контейнеризации для удаленного управления распределенной сетью ферм. | **Высокая** |

---

## 3. Решение

### Описание
Для реализации MVP утверждена декомпозиция на следующие ограниченные контексты и соответствующие им микросервисы (контейнеры):
1. **Сервис видеоаналитики (`cv_service`):** Контекст инференса моделей. Принимает RTSP-видеопоток, обрабатывает партнерскую модель на Triton Server, публикует аномалии (JSON) в локальную очередь.
2. **IoT-Шлюз автоматизации (`iot_gateway`):** Контекст взаимодействия с физическим оборудованием. Опрашивает датчики климата и регистры кормушек (MQTT, Modbus TCP), пишет метрики напрямую в локальную БД и дублирует критические события в очередь.
3. **Локальный движок правил (`local_engine`):** Ядро бизнес-логики на месте. Слушает очередь событий (Go / FastStream), сопоставляет данные (например, ИИ-детекцию со звуком визга поросят) и инициирует управляющие команды. Выступает сервером для локального Web UI оператора.
4. **Сервис синхронизации (`sync_service`):** Изолированный контекст репликации данных. Читает историю из локальной БД и асинхронно пачками (Batching) отправляет в центральную Kafka раз в 10 минут.

В качестве **локального брокера очередей** на ферме выбран **RabbitMQ** (на схеме C2 изображен в виде компактного кружка-хаба обмена сообщениями).  
В качестве **локальной СУБД** выбрана связка **PostgreSQL + TimescaleDB**.  
В качестве **среды контейнеризации и оркестрации на Edge** выбрана связка **Docker / K3s**.

*Полный исходный код контейнерных схем зафиксирован в файлах `C2_Container_V1.puml` и `C2_Container_V2.puml`.*

### Аргументация и Выбор технологий

#### Локальная оркестрация и контейнеризация: Docker / K3s


| Критерий | Обоснование выбора подхода и технологии |
| :--- | :--- |
| **Родная поддержка GPU в Docker** | Инструмент `Nvidia Container Toolkit` внутри Docker позволяет «пробрасывать» ядра локальной видеокарты хоста напрямую в ИИ-контейнер (`cv_service`) с полной утилизацией мощностей GPU и нулевой потерей перформанса на виртуализацию. |
| **Минимальный оверхед в K3s** | Полноценный Kubernetes (K8s) избыточен и «прожорлив» для Edge-сервера. K3s — это сертифицированный, сверхлегкий дистрибутив K8s, упакованный в один бинарник. Он потребляет всего **~512 МБ ОЗУ**, полностью сохраняя enterprise-стандарты оркестрации. |
| **Единый DevOps-пайплайн холдинга** | Использование K3s на фермах позволяет использовать те же Helm-чарты, манифесты и CI/CD пайплайны, что и для центрального облака «АгроПром X». Это избавляет от необходимости переобучать команду под сторонний Edge-софт. |
| **Автономность управления кластером** | K3s стабильно работает в условиях полной изоляции от интернета. При обрыве WAN-связи локальный оркестратор на ферме продолжит управлять контейнерами, перезапускать упавшие микросервисы и маршрутизировать трафик. |

#### Локальный транспорт сообщений: RabbitMQ


| Критерий | Обоснование выбора подхода и технологии |
| :--- | :--- |
| **Низкие накладные расходы** | В отличие от тяжелой корпоративной Kafka, требующей кластер Zookeeper/KRaft и гигабайты ОЗУ, RabbitMQ легковесен и стабильно работает на маломощных промышленных ПК ферм. |
| **Поддержка IoT-протоколов** | RabbitMQ из коробки через плагины поддерживает протокол MQTT. Это позволяет принимать сообщения от датчиков климата напрямую без написания дополнительных адаптеров. |
| **Гарантия доставки (ACK)** | Поддержка механизмов подтверждения сообщений и сохранения их на диск (Persistent Mode) гарантирует сохранность алертов об удушении поросят при локальных сбоях железа. |
| **Ограничение (Компромисс)** | При достижении порога нагрузки в десятки тысяч тяжелых сообщений в секунду RabbitMQ упирается в производительность CPU. Для масштаба одной фермы этот порог недостижим, однако при приеме данных со всех ферм в облаке холдинга этот брокер уступает место **Apache Kafka**. |

#### Локальная СУБД: PostgreSQL + TimescaleDB


| Критерий | Обоснование выбора подхода и технологии |
| :--- | :--- |
| **Мультимодельность** | Позволяет в рамках одной СУБД хранить реляционные профили животных (PostgreSQL) и временные ряды метрик климата и веса корма (TimescaleDB), снижая затраты на администрирование локального сервера. |
| **Автоматическое сжатие** | TimescaleDB автоматически сжимает исторические метрики на **90-95%** по колоночному принципу, решая проблему ограниченного дискового пространства на ферме. |

#### Последствия

**✅ Положительные:**
- Разделение базы данных (`local_db`) и брокера очередей (`local_queue`) гарантирует, что зависание диска СУБД при записи тяжелых аналитических логов не заблокирует мгновенную доставку экстренных алертов через RabbitMQ на GSM-модем и исполнительные реле.
- Архитектурная связка Docker Compose (на этапе MVP) и K3s (на этапе масштабирования) обеспечивает плавный, дешевый запуск продукта с гарантированной готовностью к enterprise-релизам холдинга.

**⚠️ Негативные:**
- Развертывание K3s-агентов на местах усложняет первичную конфигурацию операционной системы на физических серверах ферм.

**↗️ Зависимости:**
- Контекст видеоаналитики полностью зависит от стабильности RTSP-потоков, обеспечиваемых аппаратным шлюзом MediaMTX.

## 4. Альтернативы по интеграции сервисов


| Вариант интеграции | Плюсы | Минусы | Почему отклонен |
| :--- | :--- | :--- | :--- |
| **Синхронное взаимодействие (gRPC/REST) между ИИ и Движком правил** | Не требует развертывания брокера очередей RabbitMQ на ферме.<br>Простота отладки кода. | При зависании Движка правил ИИ-модуль заблокирует поток инференса.<br>Нет буферизации: при сбое сети алерты бесследно теряются. | Нарушает нефункциональное требование к отказоустойчивости (99,95%) и автономности контура фермы при пиковых нагрузках. |
| **Использование Apache Kafka локально на каждой ферме** | Единый стек технологий с головным офисом компании. | Чрезвычайно высокие требования к ОЗУ и процессору локального сервера.<br>Сложность кластеризации на местах. | Избыточно для объемов данных одной фермы на этапе MVP. Приведет к необоснованному удорожанию локального железа (CAPEX). |

## 5. Риски

1. **Риск переполнения локальной очереди RabbitMQ при затяжном офлайне фермы (свыше 48 часов)**
   *Меры:* Настройка политик `TTL` (Time-To-Live) для второстепенных метрик датчиков и конфигурация режима `Lazy Queues`, при котором сообщения сразу сбрасываются на жесткий диск, минуя оперативно память сервера.
2. **Риск деградации дисков (SSD/HDD) локального сервера из-за непрерывной записи логов ИИ**
   *Меры:* Сервис видеоаналитики передает в RabbitMQ только факты *изменения состояния* (триггер начала драки / триггер окончания), исключая непрерывный спам идентичными кадрами.
3. **Риск задержки алертов в альтернативном облачном решении (Вариант 2)**
   *Меры:* В Вариане 2 (Thin Edge) центральная Apache Kafka используется как высокопроизводительное вычислительное ядро (Hub), способное переваривать миллионы ИИ-событий в секунду, разделяя потоки вычислений и логики (CQRS паттерн). Обратный контур команд на кормушки при этом распределяется через облачный RabbitMQ.
