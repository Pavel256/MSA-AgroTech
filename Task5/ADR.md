# ADR-001: Проектирование мультитенантной SaaS-архитектуры, биллинга и интеграционных интерфейсов платформы «Животноводство X» (To-Be)

**Статус:** ⚠️ На рассмотрении  
**Участники:** Архитектор платформы Животноводства X, Тимлид команды Биллинга, DevOps-инженер SaaS-инфраструктуры  
**Дата:** 2026-06-06  

---

## 1. Контекст
После успешного пилотирования MVP платформы «Животноводство X» на базе распределенной топологии Heavy Edge (Умный Агент на фермах) холдинга «АгроПром X», принято стратегическое бизнес-решение вывести продукт на коммерческий B2B-рынок как SaaS-платформу для сторонних агрохолдингов. 

Переход к SaaS-модели требует кардинальной перестройки центрального облачного ЦОД холдинга. Облако должно трансформироваться в мультитенантную среду совместного использования, обеспечивая строгую изоляцию данных клиентов, гибкую монетизацию, автоматизированный онбординг внешних разработчиков и горизонтальное масштабирование ресурсов при минимизации совокупной стоимости владения (TCO).

---

## 2. Требования

### Функциональные (FURPS+)
* **Мультиарендность (Multi-tenancy):** Строгая логическая или физическая изоляция телеметрии, видеоматериалов и профилей животных между независимыми клиентами (арендаторами).
* **Гибкая тарификация:** Поддержка тарифных планов и гибридной модели оплаты на основе количества подключенных ИИ-камер и объема потребления API.
* **Панель самообслуживания (Billing ЛК):** Личный кабинет администратора клиента для регистрации, привязки карт, выбора тарифа и просмотра счетов.
* **Российский биллинг:** Интеграция с отечественными платежными шлюзами для проведения регулярных b2b-списаний.
* **Открытый API для клиентов:** Предоставление защищенного партнерского API для интеграции платформы животноводства с собственными ERP-системами (1С) и BI-платформами сторонних агрохолдингов.
* **Автоматизированный онбординг:** Доступ к Порталу для разработчиков (Dev-портал) с интерактивной документацией OpenAPI/Swagger и механизмами самостоятельной генерации API-ключей.
* **Сбор ML-датасетов:** Централизованный сбор дедуплицированных и полностью обезличенных данных со всех ферм для обучения собственных ML-моделей компании.

### Нефункциональные
* **Безопасность (Data Isolation):** Исключение любых рисков утечки данных одного агрохолдинга к другому.
* **Горизонтальное масштабирование:** Способность облачного API Gateway и очередей Kafka выдерживать линейный рост нагрузки при подключении новых компаний.
* **Ограничение квот (Rate Limiting):** Защита API от перегрузки запросами со стороны ERP-систем отдельных клиентов.

## 3. Задача 1: Проработка вариантов изоляции данных в SaaS

Для центрального хранилища данных (DWH ClickHouse) и бэкенда в облаке проанализированы три архитектурных подхода к изоляции данных арендаторов (Tenants):

### Сравнительный анализ моделей изоляции данных



| Модель изоляции | Плюсы | Минусы | Почему отклонен / Принят |
| :--- | :--- | :--- | :--- |
| **Вариант А: Физическая изоляция (Database-per-Tenant)**<br>У каждого клиента свой физический сервер или выделенный инстанс БД ClickHouse.<br>*(Схема: `C2_isolation_db.puml`)* | Максимальный уровень безопасности.<br>Простота резервного копирования отдельного клиента.<br>Отсутствие эффекта "шумного соседа". | Высокая стоимость инфраструктуры (высокий TCO).<br>Сложность администрирования и миграции схем при большом количестве клиентов.<br>Невозможно агрегировать данные для обучения общих ML-моделей. | **Отклонен** для MVP SaaS-платформы из-за высокой стоимости поддержки и противоречия бизнес-цели минимизации TCO. Может предоставляться как VIP-опция для Enterprise-клиентов. |
| **Вариант Б: Изоляция на уровне схем (Schema-per-Tenant)**<br>БД общая, но у каждого клиента свое изолированное пространство имен/схема внутри ClickHouse.<br>*(Схема: `C2_isolation_schema.puml`)* | Хороший баланс стоимости и изоляции.<br>Права доступа разграничиваются на уровне СУБД для каждого пользователя схемы. | ClickHouse исторически не оптимизирован под поддержание тысяч отдельных схем и таблиц (раздувается метаинформация в ZooKeeper/KRaft, падает перформанс). | **Отклонен** из-за технических ограничений выбранной DWH-технологии (ClickHouse) при долгосрочном масштабировании на большое количество арендаторов. |
| **Вариант В: Логическая изоляция в общих таблицах (Shared Database + Row-Level Security)**<br>Таблицы общие. В каждой строке присутствует колонка `tenant_id`. Фильтрация данных происходит на уровне движка СУБД через политики RLS.<br>*(Схема зафиксирована в финальном To-Be решении)* | **Минимальная стоимость владения (Низкий TCO).**<br>Максимальная скорость выполнения аналитических запросов.<br>Идеально для агрегации обезличенных датасетов для обучения собственных ML-моделей. | Риск ошибок в коде бэкенда, которые могут привести к утечке данных (нивелируется использованием встроенных политик ClickHouse Row-Level Security). | **ПРИНЯТ** в качестве целевого стандарта для SaaS-платформы как наиболее экономически и технически эффективный вариант, обеспечивающий горизонтальное масштабирование ClickHouse. |

## 4. Задача 2: Разработка подсистемы биллинга и монетизации

В архитектуру центрального SaaS-облака внедрены два новых специализированных контейнера:
1. **Сервис биллинга и подписок (`billing_service`):** Отвечает за хранение бизнес-правил тарификации, ведение балансов личных кабинетов клиентов, обработку тарифных планов и формирование платежных документов.
2. **Сервис сбора метрик использования (`metering_service`):** Легковесный демон на Go/Python. Он асинхронно вычитывает из центральной Kafka логи активности умных агентов и фиксирует: а) количество активных ИИ-камер на каждой ферме; б) объем входящих REST API запросов от систем клиентов. Собранные сырые метрики использования (Usage-data) агрегируются и передаются в `billing_service` для тарификации по модели *Pay-as-you-go*.

### Интеграция с российскими платежными системами
Для проведения расчетов B2B-монетизации на российском рынке выбрана интеграция с платежным шлюзом **ЮKassa / Т-Бизнес** по следующей схеме:
* При онбординге в ЛК Самообслуживания Администратор клиента привязывает корпоративную банковскую карту или расчетный счет.
* `billing_service` через защищенное HTTPS-соединение инициирует привязку метода оплаты (`Recurrent Payments`) и сохраняет обезличенный маскированный токен (клиентский ID платежного шлюза). Физические данные карт внутри SaaS-платформы не хранятся (выполнение стандартов безопасности).
* В конце расчетного периода (раз в месяц) `billing_service` запрашивает у `metering_service` объем потребления, рассчитывает сумму и отправляет в платежный шлюз команду на автоматическое безакцептное списание.
* Статусы успешности списаний поступают в API Gateway платформы через систему `Webhooks` от платежного шлюза, автоматически продлевая подписку токенов Умных Агентов в Keycloak.

## 5. Задача 3: Проработка интеграционных интерфейсов с клиентами (B2B API)

Для внешних систем клиентов (1С, Power BI, ERP) SaaS-платформа предоставляет выделенный слой защищенного **B2B REST API**.

### Scope (Границы) доступного функционала внешнего API:
* **Слой Чтения (Телеметрия):** Выгрузка исторических графиков климата, объемов фильтрации воды, веса и расхода корма в формате JSON/CSV для импорта в BI клиентов.
* **Слой Учета (События):** Получение реестра зафиксированных инцидентов (количество драк, факты гибели или падежа, подтвержденные риски задавливания поросят с привязкой к ID загона) для автоматического списания или проводки документов в 1С:Агро клиента.
* **Слой Конфигурации:** Чтение и удаленное изменение порогов срабатывания датчиков (например, удаленное изменение лимита температуры в секции из центрального офиса клиента). Прямое управление реле исполнительных механизмов через внешний API **блокируется** из соображений биобезопасности ферм.

### Документация и Онбординг (Dev-портал)
Для реализации концепции самостоятельного быстрого онбординга в облачный контур внедряется **Портал для разработчиков (Dev-портал)** на базе open-source решения **Backstage**:
* Доступ на портал предоставляется ИТ-инженерам клиентов автоматически после активации подписки в Личном Кабинете.
* Dev-портал содержит интерактивную документацию формата **OpenAPI (Swagger)**. Клиент может прямо из браузера протестировать запросы к API животноводства в изолированной песочнице (Sandbox).
* Через Dev-портал ИТ-специалисты клиента самостоятельно генерируют долгоживущие токены доступа (`API Keys / Bearer Tokens`) с привязкой к своему `tenant_id` и настраивают политики ограничения квот (`Rate Limiting`), защищая свою интеграцию от перегрузок.

## 6. Задача 4: Финальная архитектура To-Be и примеры интеграции

### 1. Финальные архитектурные схемы SaaS-платформы (To-Be)
* Визуализация контекста всей экосистемы (C1) находится в файле: `C1_saas_platform_to_be.puml`
* Детальная контейнерная структура микросервисов, биллинга и хранилищ (C2) находится в файле: `C2_saas_platform_to_be.puml`

### 2. Пример интеграции холдинга «АгроПром X» в качестве клиента SaaS
После коммерческого запуска платформы текущая компания «АгроПром X» мигрирует со своей обособленной структуры и становится рядовым арендатором (Tenant) на единой SaaS-платформе. 
* Локальные Умные Агенты (Heavy Edge) на фермах компании продолжают работать автономно, но теперь при асинхронной отправке отчетов в облако раз в 10 минут они принудительно подставляют в заголовки токен авторизации и параметр `tenant_id = 'agroprom_x'`.
* Существующие корпоративные сервисы компании (Реакт-портал, Power BI, аналитический модуль Spark) больше не обращаются напрямую в базы данных, а переключаются на чтение данных через единый открытый **SaaS API Gateway** по стандартным Bearer-токенам, проходя фильтрацию данных Row-Level Security в общем ClickHouse DWH.

*Детальная контейнерная схема этой миграции и интеграции зафиксирована в файле:* `C2_client_integration_agroprom_x.puml`

## 7. Риски и компромиссы SaaS To-Be решения

1. **Риск "Шумного соседа" в общей базе ClickHouse**
   * *Описание:* Один крупный клиент может запустить тяжелый BI-отчет и заблокировать потоки чтения общего ClickHouse, из-за чего веб-порталы мелких клиентов начнут выдавать ошибки по таймауту.
   * *Меры:* Внедрение механизма квотирования ресурсов (Max Execution Time, Max Threads per User) на уровне политик профилей пользователей в ClickHouse DWH для каждого `tenant_id`.
2. **Риск компрометации данных (утечка данных между конкурентами)**
   * *Описание:* Ошибка в коде общего API-шлюза может привести к тому, что Агрохолдинг А увидит статистику Агрохолдинга Б.
   * *Меры:* Внедрение сквозного тестирования (Integration Security Tests) на этапе CI/CD. Включение механизма ClickHouse Row-Level Security (RLS) на уровне таблиц базы данных, когда сама СУБД отсекает строки, не соответствующие `tenant_id` текущей сессии Keycloak.
3. **Риск неоплаты подписки при сохранении локальной автономности**
   * *Описание:* Клиент перестал платить за SaaS-тариф, но его Умный Агент на ферме продолжает работать в офлайне.
   * *Меры:* Умный Агент запрашивает проверку лицензионного ключа в облаке раз в 10 минут во время асинхронного батчинга метрик в Kafka. Если баланс клиента отрицательный более 7 дней, центральное облако отправляет на Агент команду блокировки UI. Локальный оператор на ферме теряет доступ к Веб-интерфейсу настроек кормушек и графикам. При этом критический контур безопасности продолжает работать принудительно, чтобы исключить физическую гибель животных и юридическую ответственность SaaS-платформы за падеж.
4. **Риск "Холодного старта" ML-моделей для новых клиентов**  
   * *Описание:* Новые клиенты при онбординге в SaaS ожидают мгновенной точности ИИ, однако из-за уникальных ракурсов камер, высоты подвеса или другой породы животных партнерская модель может начать сыпать ложными срабатываниями. Это грозит оттоком клиентов.  
   * *Меры:* Переиспользование облачного контейнера `ml_train_module` (Python + Spark) для сбора дедуплицированных сырых кадров ошибок из MinIO всех текущих клиентов. Внедрение механизма дообучения моделей: новый клиент первые две недели работает в "Режиме обучения", система собирает видеопоток в Data Lake, автоматически дообучает нейросети под ракурсы конкретной фермы и активирует боевой контур только после достижения точности > 92%.

5. **Риск "Сетевого шторма" (DDoS) со стороны Умных Агентов при восстановлении связи**  
   * *Описание:* При затяжном обрыве интернета на фермах крупные клиенты накопят гигантский объем телеметрии в своих PostgreSQL/TimescaleDB. В момент восстановления связи фермы начнут слать миллионы строк в облако, что создаст непреднамеренную DDoS-атаку на API Gateway и забьет буфер центральной Kafka.  
   * *Меры:* Внедрение алгоритма экспоненциального затухания с джиттером в микросервис `sync_service` на Умном Агенте. Сервер фермы отправляет накопленные за время офлайна исторические данные случайными небольшими пачками с рандомной задержкой в диапазоне от 1 до 10 минут, плавно распределяя нагрузку на облачный API Gateway и центральную Kafka.
